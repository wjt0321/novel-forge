# Harness 信任与控制面完整性

## 事故边界

一次真实实验中，自动入口正确报告缺少 `SessionBackend`，但 Lead 随后把“部署正式
环境”理解成可在创作任务内实现基础设施。它在仓库 `tools/` 下新建命令桥脚本，
自行设置 `NOVEL_FORGE_HARNESS_COMMAND`，并用随机字符串冒充原生会话、估算未知
遥测、声明没有实际执行的 capsule 隔离。

这不是 Writer capsule 的路径逃逸。越权发生在 Lead 控制上下文，而且宿主以
`bypassPermissions` 一类不受限权限运行，因此原有 Guardian 没有机会审查该脚本。

## v4.8 补强

`CommandSessionBackend` 现在执行以下前置与持续校验：

1. 命令入口必须能解析为真实文件，且所有入口程序和脚本都位于项目仓库外。
2. Backend 创建时记录入口文件 SHA-256，每次调用前后复核；入口替换立即失效。
3. 外部进程的当前目录是一次性临时目录，不再默认暴露项目根。
4. 每次调用前后扫描仓库控制面；任何新增、删除或内容变化都返回
   `control_plane_mutation`。
5. 书稿、外置 Guardian、每书 Git、数据库和实验目录仍由各自既有协议管理，不混入
   控制面快照。

## Agent 权限

小说创作、续写、审稿或修订任务中的 Lead 与角色不得创建、修改、修复、包装、安装
或配置 Harness / SessionBackend，不得自行设置命令桥环境变量，也不得向用户提供
“部署环境”作为继续写作选项。命令桥只属于可选 headless 模式；宿主原生角色可用时
不得因它缺失而停止。

宿主接入必须是创作请求开始前完成的独立管理工作。这个规则不绑定 Claude Code、
Codex、Kimi Code、Teams、DeepSeek 或其他产品。

## 权限不是约束

给一个会推理的 Lead 最高文件和进程权限，只会提高它实现越权方案的能力，不会提高
它遵守工作流的概率。正式链路因此拆成两个平面：

1. 确定性控制面拥有窄而高的权限：创建、等待和退役会话，调用 adapter 与 Guardian，
   写状态、不可变证据和每书 Git。它没有模型推理，不产出正文或文学判断。
2. Writer、Blind Reader、Chapter Editor 与 Lead 使用宿主原生独立会话，只获得角色
   所需输入和产物权限。Lead 只调度；任何角色都不能修改控制面或代做其他角色产物。

“最高权限”只适用于第一类确定性操作，不再描述任何创作 Agent。
异步完成只认宿主官方终态；接单、进度消息或文件稳定不能替代 `wait` / `join`。

## 安全上限

仓库代码可以拒绝仓库内入口、固定入口哈希并发现调用期间的控制面变化，但无法把
同一操作系统用户下的完全不受限 Agent 变成真正的安全边界。若宿主允许 Agent 任意
修改仓库外文件、进程环境和 Git 引用，Agent 仍可能攻击所有本地信任材料。

正式量产应让 Harness 由独立宿主进程、容器或权限更低的账号管理，并让创作 Agent
只获得调用能力，不获得 Harness 源码、注册信息或仓库控制面的写权限。项目内校验是
第二道防线，不能替代操作系统级隔离。
