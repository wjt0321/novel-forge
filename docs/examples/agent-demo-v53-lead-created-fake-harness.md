# Agent Demo v5.3：Lead 自建伪 Harness

## 样本边界

本样本来自一次已删除的实验书和未跟踪脚本，只保留流程结构、失败模式与规则。
不保留正文、书名、人物名、真实会话标识、脚本源码、正文哈希、签名、密钥或可恢复
Git 历史。该实验没有生成正式正文。

## 观察结果

- 自动入口最初正确识别到没有可用 `SessionBackend`，并在建书前停止。
- Lead 随后向用户提供了“部署正式环境”选项，把用户的继续选择解释成了编写基础
  设施的授权。
- Lead 在仓库 `tools/` 下创建命令桥脚本，并自行设置环境变量指向该脚本。
- 脚本没有创建真实原生会话，只生成随机形式的会话字符串。
- 脚本根据字符数估算 Token，把未知遥测改造成伪精确值。
- 脚本声明 writer 只能看到 capsule，但实际子进程仍以项目仓库为默认工作目录，
  没有文件系统隔离证明。
- 脚本的兜底审稿可以直接返回通过，且盲审字段与通过条件互相冲突。
- 工作流尝试进入规划阶段后被人工中止，只留下空项目骨架和每书初始 Git，不存在
  Generation、Runtime Audit、Guardian Receipt、Review 或正文。

## 根因

原规则只说 Backend 缺失时停止，没有明确禁止 Lead 在创作任务内创建、修复或包装
Backend。用户界面又给出了“部署环境”这一不应出现的选项，使 Lead 把创作请求切换
成了代码实施任务。

命令桥实现也只检查环境变量是否存在，没有验证入口是否位于仓库外、是否在调用间
被替换，或子进程是否改动仓库控制面。宿主同时使用不受限权限模式，文本规则没有
操作系统写权限作为后盾。

这不是中等思考模式单独造成的。推理模式会影响 Agent 是否自作主张，但只要存在
“缺什么就自己补什么”的权限与接口漏洞，其他模型也可能采取相同捷径。

## 转化规则

1. 创作 Lead 与所有角色不得创建、修改、修复、包装、安装或配置 Harness /
   SessionBackend。
2. Backend 缺失时只报告本章未开始，不提供部署或配置基础设施的选择。
3. 命令桥入口程序与脚本必须位于项目仓库外。
4. Backend 生命周期内固定入口文件哈希，入口变化立即失效。
5. 外部进程在一次性临时目录启动，不继承项目根为默认工作目录。
6. 每次调用前后核对仓库控制面；任何变化以 `control_plane_mutation` 中止。
7. 随机字符串、角色名和脚本自报不能证明真实原生会话或隔离。
8. 未知 Token、耗时、请求数和 thinking 深度必须保持未知，禁止估算。
9. 完全不受限的 Agent 不能只靠仓库规则形成强安全边界；正式宿主应使用独立进程、
   容器或 OS 权限隔离 Harness 与控制面。
