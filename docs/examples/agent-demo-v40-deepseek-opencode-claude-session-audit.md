# DeepSeek + OpenCode / Claude Code 五章实验审计

## 结论

这次实验再次说明：问题不是某一家模型“不会写”，而是让一个原生 writer session
连续承担五章正文、工具调用、审稿和证据维护后，文学惯性、上下文缓存和流程债务会一起
累积。两套 Harness 的速度、调度方式和失败形态不同，但都没有把 v3.9 已声明的
“每章独立会话”真正执行成会话边界。

因此 v4.0 采用厂商无关的章节序列协议：

- 一章只能由一个独立原生 writer session 承担；
- 上一章完整 `ready` 后，编排器才获得下一章的 launch directive；
- 下一章必须 claim 一个此前未使用的真实 session ID；
- 连续请求默认 1 章，自动批量最多 4 章；5 章请求必须拆分；
- 连续性来自 Canon、承诺、上一章末段、Voice exemplar 与本章 scene package
  组成的有界交接包，不来自旧对话“记得”；
- 每章 2,000,000 cached-input tokens 是止损硬上限，不是应当吃满的配额。

具名模型和 Harness 只用于提供实验样本，协议没有针对 OpenCode、Claude Code 或
DeepSeek 写特例。

## 脱敏运行数据

| 样本 | 用时 | assistant 消息 | 工具调用 | cached input | 结果 |
|---|---:|---:|---:|---:|---|
| OpenCode | 31 分 37 秒 | 184 | 254 | 21,075,584 | blocked |
| Claude Code | 75 分 34 秒 | 605 | 312 | 95,208,320 | blocked |

OpenCode 原始五章合计约 34,489 CJK。前四章接近最低字数，到了第五章突然扩张到
11,111 CJK，并出现 9 个 `not-is-flip` blocking。它很快，但后段已经表现出赶工、
格式化收束和审计缺失。

Claude Code 当前五章合计约 61,508 CJK，第三至第五章依次扩张为 9,555、20,695、
15,761 CJK。会话在末段明确跳过部分正式审稿记录；第四章 blind-reader 仍为
`uncertain`，第五章未完成审稿闭环。长思考没有换来更稳定的收敛，反而让缓存输入
增长到约 9,521 万。

## 路径污染

Claude Code 的子 writer 曾把第二章写入 OpenCode 项目，随后主会话又把这份错误项目
正文复制进自己的项目。审计时两边当前第二章 SHA-256 完全相同：

`68f8378f33f520c7e79d6da91bcf0dd3ac99a78023342c3c28962145ac4587b4`

OpenCode 工具日志中可恢复的原始第二章则为 5,843 CJK，SHA-256 为：

`ee6df355eed2980507eec6178bf4e406e7d750294b8d147d3c5861b8866620b9`

所以当前第二章不能参加文学横向比较。这个故障也证明“子 Agent 名称不同”不等于
执行环境隔离，章节序列必须同时绑定书 slug、章节号、交接包哈希和真实 writer
session ID。

## 文学判断

第一至第二章仍能看到模型第一次落笔时对场景压力、物件和人物动作的主动选择。进入
长会话后，表达开始服务于模型已经形成的局部套路：固定动作承担紧张、固定物件承担
收束、短句承担高潮、感官细节承担“人物在场”的证明。章节越往后，文字越多，但新的
人物判断和不可逆变化没有等比例增加。

一章一会话不是把长篇拆成互不相干的短篇。它切断的是模型上一轮生成时形成的低层
口头禅；作品身份则由 Voice exemplar、Canon、人物知识状态、开放承诺和上一章末段
继续传递。长会话依赖模型记得，章节化生产依赖系统证明它没有忘。
