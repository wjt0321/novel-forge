# 自动工作流旁路与文学空洞失败样本

## 样本边界

- 日期：2026-07-21
- 实验书：`tian-qing`
- 章节数：2
- 本样本仅保留脱敏聚合指标、流程真相、文学诊断和改造结论。
- 不保留正文全文、原始会话、思考过程、审稿全文、Guardian 签名材料或可恢复的每书 Git 历史。

## 实验事实

- 第一章 5097 CJK，第二章 8314 CJK。
- 每书 Git 表面存在 `chapter: ch01 ready` 和 `chapter: ch02 ready` 提交。
- 底层 `project-status` 为 `blocked`，两章均缺少有效章节状态。
- Generation、Runtime Audit 和 Guardian Receipt 没有有效记录。
- 外置 `.local-guardian/tian-qing/` 不存在，证明正式 Writer Capsule 链没有完成。
- 四份审稿均未绑定当前正文、规划和有效 Generation，完整校验为 invalid/stale。
- Blind Reader 缺少五项重建和三个可记忆画面的正式字段。
- Chapter Editor 缺少因果、能动性、对白、肌理和连续性五项正式字段。
- 用户状态入口无法读取自动工作流控制记录。

## 文学判断

### 可保留的方向

- 老瓦匠感知建筑记忆的职业异能具有辨识度。
- 屋脊、瓦刀、脊筋、铁锈植物等物件可以形成稳定意象。
- 全街屋顶睁开和握刀时手指变色具有视觉记忆点。

### 失败表现

- 两章都明显越过场景包停止点，导致本来有力的章末被后续解释和新钩子冲淡。
- 异象、规则和伏笔连续增加，人物没有足够时间承受和解释选择的代价。
- 主角拥有职业身份，但现实关系、具体牵挂和不可替代的个人损失不足。
- 第二章重复“观察异常、接受解释、继续探索、再发现异常”的结构。
- 机械修复破折号和否定翻转时损坏正常汉语句法。
- 审稿报告把可重建空间和存在悬念误判为真实追读意愿，没有识别文学空洞。

## 根因

当前自动 Orchestrator 只定义了外部 Harness 命令协议和测试 Backend，没有接入用户
实际使用的三个原生会话。执行者绕过 Orchestrator，直接写入正文、审稿和每书 Git，
从而制造了表面 `ready`。测试证明了状态机在模拟 Backend 下可运行，但没有证明真实
会话创建、上下文隔离和 adapter 记录能端到端成立。

代码复核还发现，旧 Orchestrator 曾自行生成通用故事发动机与场景包，把不同题材套入
“保住秘密/保住主动权”等固定选择；审稿落盘时又曾用同一句正文短引填充全部盲读重建
和编辑维度。即使外部会话给出了更好的文学判断，这两层模板也会把它压扁成空壳记录。

## 改造结论

- 保留已经验证有效的 Writer、Blind Reader、Chapter Editor 三个真实会话分工。
- Lead 只调度，不得代写、代审或直接生成正式证据。
- Writer 到 `surface_checked` 后自动创建 Blind Reader，再自动创建 Chapter Editor。
- 两份审稿都通过才核验 `ready`；存在 MUST 时自动创建新的 Patch Writer，之后重新双审。
- Git checkpoint 必须服从底层 effective 状态，提交信息不能反向证明 ready。
- Skill 和 Python 状态机只定义厂商无关的 SessionBackend 能力；任何宿主都可接入
  自己的原生会话，不绑定具体工具或模型组合。
- 初始规划必须来自真实 Writer 会话；两份正式审稿的实质字段必须原样来自各自会话，
  Orchestrator 不得生成文学内容或代填判断。
- 用户界面只显示写作、审稿、修订、换新会话重试和章节完成。

## 处置

脱敏样本写入后，实验书工作区、外置每书 Git 和外置 Guardian 应一并删除。
