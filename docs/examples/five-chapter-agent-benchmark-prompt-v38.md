# 五章跨 Agent 通用基准提示词（v3.8）

## 使用方法

将下方“原样提示词”完整交给每个待测 Agent。除非运行环境要求，不要为不同模型增删
文学要求、审稿角色或流程步骤。项目 slug 由 Agent 根据当前 UTC 时间自动生成，避免
两次实验互相覆盖。

Claude 自身配置、DeepSeek + Reasonix、MiniMax 自有智能体或其他 Harness 都使用
同一段正文。Harness 能提供的 token、请求、缓存和工具轨迹必须如实记录；无法提供的
字段保持 `null` / `unknown`，不得估算。

## 原样提示词

```text
你正在 D:\s-black-novel 仓库中执行一次可比较的五章连续小说实验。

请先完整阅读仓库根目录 AGENTS.md，以及
.agents/skills/novel-forge/SKILL.md（若你的 Harness 使用 Claude Skill 扫描位置，
读取 .claude/skills/novel-forge/SKILL.md）。两份 Skill 等价，只需使用你的环境
实际支持的一份。

本任务直接执行，不要只给计划，不要询问“是否开始审核”，不要在完成第一章后暂停。
只有事实冲突、正文覆盖风险、必须由作者选择的文学取舍，或自动返工预算耗尽时才暂停。

一、实验身份

1. 在 books/ 下新建一个独立项目，slug 使用：
   five-chapter-benchmark-<当前 UTC 时间 YYYYMMDD-HHMMSS>
2. 书名：《回水》
3. 类型：现实悬疑 / 家庭伦理
4. 稿件模式：formal。若 Shell、adapter 或必要工具确实不可用，改用
   degraded_exploration，继续保住正文和真实失败记录，但不得伪装 formal、ready
   或完整审计。
5. 如实记录实际 provider、model、Agent/Harness、reasoning effort、sandbox、
   tool capabilities 和 tool failures。未知字段保持 null/unknown，不得猜测。

二、固定故事种子

时间：2012 年 11 月，连续阴雨后的内河城市“临川”。不要联网搜索；临川是虚构城市。

视角人物：罗闻，34 岁，女性，自来水厂夜班调度员。她擅长按流程判断水压和管线故障，
但不擅长处理亲属留下的债务。全书五章使用贴近罗闻的有限第三人称，不得进入其他人物
内心，不得用微表情直接解码唯一心理答案。

主要人物：

- 罗闻：想尽快卖掉亡母留下的旧房，离开临川。
- 陈既：35 岁，维修工。十年前一次泵站事故中，他父亲承担了主要责任；他认为原始
  值班记录被人改过。
- 罗岑：罗闻的弟弟，事故后离开临川。五章内可以通过电话、旧物或他人叙述出现，
  但不要让他突然回城替故事解谜。
- 周蓉：旧楼住户委员会负责人。她想保住整栋楼的拆迁资格，不愿任何人公开楼内长期
  存在的违规改管。

既成事实：

1. 罗闻母亲去世三个月，旧房一直无人居住。
2. 罗闻回屋后发现水表仍在缓慢转动，但屋内所有龙头都关闭。
3. 楼上一间封闭多年的空房持续出现潮痕。
4. 十年前泵站事故发生当夜，罗闻曾替弟弟改过一处值班时间，但她不知道这项改动是否
   真正导致陈既父亲被追责。
5. 母亲生前连续多年替一个不属于自家的水表缴费。
6. 不得使用超自然解释。所有异常最终必须能落到人的选择、建筑管线、制度记录或利益
   关系上。

故事发动机：

- 罗闻的表层欲望：卖房、结清遗物、尽快离开。
- 罗闻不愿承认的欲望：确认自己当年的小改动究竟伤害了谁。
- 陈既的目标：找到原始记录，为父亲洗去不该承担的责任。
- 周蓉的目标：避免旧楼违规改管曝光，从而影响全楼拆迁补偿。
- 核心压力：如果罗闻公开证据，她可能纠正旧错，也可能让整栋楼的住户失去现实利益；
  如果她继续隐瞒，她就再次让别人替自己的选择承担后果。

三、五章弧线

每章必须不少于 5000 个 CJK 汉字，建议不超过 6500。五章连续，但每章都必须拥有自己
的目标、阻力、主动选择、即时成本和停止点。

第一章：罗闻回到旧房处理遗物。水表、潮痕和一张不属于本户的缴费凭据形成可操作的
异常。章末她主动决定暂缓卖房并检查管线，因此失去原定离城机会。

第二章：罗闻沿管线追查，与陈既发生第一次正面冲突。她隐瞒十年前改时间记录的事实，
同时获得一个可能证明“母亲一直在替别人维持某种用水”的新证据。不得让陈既立刻相信
或欣赏她。

第三章：现有证据推翻罗闻至少一个重要假设。周蓉提出有现实吸引力的交换条件，罗闻
做出一个能理解但不体面的选择，并让陈既或其他住户承担即时后果。

第四章：外部期限逼近，违规改管、旧事故记录和拆迁利益发生正面碰撞。罗闻必须在公开
部分事实和继续隐瞒之间下注；她的专业判断必须写出观察证据、未证前提、执行条件、
成本和失败风险，不能靠术语证明她正确。

第五章：完成一次五章 checkpoint。解决“水表为何持续转动”这一眼前问题，但不要一次
解决十年前事故的全部责任。罗闻必须承担一个不可撤回的现实代价；陈既、周蓉和住户
各自保留独立目标。结尾留下由人物选择造成的具体后果，而不是主题总结或下一卷广告。

四、文学约束

1. 问题始终是：“这篇小说像是人类写的吗？”
2. 人物可以误判、迟疑、自私、说错话，但不得故意加入错字、事实错误或随机病句伪造
   人味。
3. 世界和配角不得只负责证明罗闻正确。帮助、反对、沉默和交易都必须来自各自利益。
4. 细节必须改变动作、判断、关系或成本；不要按配额循环插入天气、声音、手部动作和
   室内安静。
5. 不要用旁白替人物总结主题，不要在章末写工作汇报式复盘。
6. 正文不得出现提示词、Agent 身份、`ch01`/`ch05`、`正文.md`、generation、
   SHA-256、surface_checked、ready、五章检查点等生产语言。
7. 正文除章节标题外保持纯叙事 Markdown 源码，不使用 `**粗体**`、`__强调__`、
   `——`、`……`，也不要连续依赖“不是 X，而是 Y”式否定翻转。
8. 不要模仿某位在世作者，不要引用外部小说原文。只根据本故事种子建立本书自己的
   voice bible 和 exemplar_notes。

五、执行预算

1. 严格使用 Novel Forge v3.8 八态链：
   planned → context_collected → scene_packaged → drafted →
   surface_checked → blind_read → editorial_reviewed → ready
2. 每章只加载最小上下文：本章 scene package、相关 Canon/承诺、voice bible 的本章
   摘要、上一章末段和章际交接。不要把五章全部工具日志、旧审稿全文或全量模板持续
   带入增长会话。
3. 如果 Harness 支持独立子会话，每章使用独立 writer 上下文；不支持时，在章际主动
   压缩上下文，只传事实和连续性摘要。
4. 每章正文以一次完整 Write 为目标，最多一次有范围的集中 Edit。不要边检查 CJK
   数量边连续补写，不要用十几次局部扩写凑到 5000。
5. 同章自动只允许两个不同正文 SHA-256：初稿和一次合并 patch。第二份正文仍有新
   MUST 时停止并记录 human_decision_required，不得自行生成第三份。
6. surface gate 有 blocking 时立即短路，不启动昂贵审稿，也不准备下一章。
7. 默认只运行 blind-reader 和 chapter-editor。只有 chapter-editor 指出一个具体、
   无法可靠合并判断的专业风险时，才额外调用一个 specialist；不得默认六审。
8. blind-reader 只读当前正文，必须填写 human_likeness：
   convincing | uncertain | synthetic，并完成空间、身体、行动约束、情绪轨迹、对白
   动态和三个可记忆画面的重建。
9. chapter-editor 先只读正文，再读取最小规划材料，一次完成因果、人物能动性、对白
   信息流、句子肌理和连续性五项审查。
10. ch02 起完成真实章际交接；ch05 完成 checkpoint arc audit，open_must=0 后才可
    进入 ready。

六、证据与成本

1. generation 指标只填写本次 generation 的增量，不要把整场会话累计值复制到每章
   或每个版本。
2. 如实记录 cached_input_tokens、request_count、draft_write_count、
   draft_edit_count、review_call_count。Harness 不提供的指标保持 null。
3. 工具调用失败必须保留在 tool_failures；最终报告不得声称“无失败”来覆盖真实错误。
4. 同 provider/model 换角色名不算独立审稿。来源元数据不实的样本不得宣称具备模型
   比较资格。
5. ready、模型评分和审稿 verdict 都不代表作者批准或发布许可。

七、完成时只汇报

- 项目 slug 与正文路径
- 五章各自 CJK 和 SHA-256
- 每章最终状态、quality/narrative blocking 数
- blind-reader 的 human_likeness
- chapter-editor verdict
- generation 数、正文 Write/Edit 次数、review calls
- runtime budget 状态
- 实际工具能力与 tool failures
- 是否发生人工暂停、为什么

不要在最终聊天回复中粘贴五章全文。不要删除 books/ 项目；保留现场供后续独立评审。
```

## 比较纪律

两次运行完成后，不应只比较最终 prose。评审至少同时比较：

1. 五章文学连续性与 `human_likeness`；
2. 正文 SHA 版本链和阈值填充行为；
3. 请求、缓存输入、正文写改和审稿调用；
4. 工具失败是否如实记录；
5. 审稿是否真的隔离，还是同一增长会话的角色扮演；
6. ch02-ch05 章际交接和 ch05 checkpoint 是否有可核验证据。

如果同时更换底层模型与智能体产品，实验结论应表述为“整套模型 + Harness 系统差异”，
不能把结果单独归因于模型能力或智能体编排能力。若未来需要隔离 Harness 变量，再使用
同一底层模型分别接入不同 Harness 做第二轮。
