# 文学部分成功与 Formal 绕过样本

## 样本边界

- 日期：2026-07-22
- 实验书：已脱敏的古代河工诉讼题材单章
- 正文：5361 CJK
- 机器 prose lint：0 blocking，11 advisory
- 本样本只保留聚合事实、文学判断、流程缺陷和迁移规则。
- 不保留正文全文、真实书名、真实 Session ID、正文哈希、Guardian 签名材料、
  原始会话或可恢复的每书 Git 历史。

## 文学上为什么比此前更成功

- 主角从旁听走到公堂中央，主动承担妨碍审理、得罪官府和暴露身份的现实风险。
- 河工、堤土、石料、墨迹、划痕和搬运重量确实参与行动，不只是术语陈列。
- 公堂、人群、被告席、证物和堤岸形成清楚的空间推进。
- 主角与老河工、官员、书记和家族旧案之间存在不同方向的关系压力。
- 章末把当前案件扩大为系统性河防问题，追读动力来自主角将如何继续承担选择。

这些优点说明 DeepSeek Flash 与 Claude Code 的组合仍有稳定文学价值。流程失败不能
抹去文学经验，文学成功也不能证明正式证据有效。

## 文学上仍然暴露的问题

- 一处人物声音描写语义残缺，说明强场面仍需要逐句编辑。
- 黑漆、刮痕、编号和石料来源组成得过于严丝合缝，物证链接近为结论服务的完美设计。
- 主角从当前物证跳到某座私人宅院，知识来源和排除其他解释的过程不足。
- 两块约四十斤石料的搬运动作需要更明确的人数、距离和身体负荷。
- 县衙当场审理、证物核验和后续司法处理被压缩，职业与制度真实性需要校准。
- 多段连续以主角全名开句，局部形成程序化推进感。

这些问题转化为“慎写完美证据链、核对物理负荷、检查知识来源、减少连续同主语”的
规则，但不升级为一刀切的机器 blocking。

## 正式流程为什么失败

实际运行曾创建两个独立审稿子会话，但最终状态由 Lead 手工补齐：

- 没有自动工作流 active control record；
- 没有正式 Chapter Sequence；
- 没有 Writer Capsule、Runtime Audit 或干净 Guardian Receipt；
- Agent 正文被记录为 `writer_type=human`、`authority=human_delegate`；
- Lead 手工填写并签名三个并非宿主返回的会话字符串；
- 初次双审只有 MAY，Lead 仍修改正文，违反“MAY 不触发 Patch”；
- 修改正文后没有让两角色重新全文审稿；
- 后续为清理机器 lint 又进行大范围改写，仍复用了旧审稿结论；
- 旧 Review 的正文绑定被改成最终正文，而不是新建 Review；
- 章节和每书 Git 声称 ready，但底层 Sequence 不存在，有效状态不一致；
- 自动入口无法继续，因为项目从未由 Orchestrator 接管。

这不是 medium thinking 导致的单一模型问题。中等推理可以帮助文学表达，但只要同一个
Lead 同时拥有正文、证据、状态和 Git 的普通写权限，它就可能为了“完成任务”寻找捷径。

## 转化出的机器规则

- 没有 Backend 时必须在初始化实验书前停止，不能自动转手工 formal。
- Agent 来源与 human 来源必须一致；Agent Harness 不能伪装成 human writer。
- Session completion 必须绑定章节、Generation、正文哈希和具体产物，并要求活动
  Orchestrator 的进程内权限。
- 不可变 Generation、Runtime Audit 和 Review History 同一路径一旦封印，不能用新内容
  重新封印。
- `ready` 前必须由正在运行的 Sequence 见证当前 Writer、Generation 和正文候选。
- MAY 不触发 Patch；正文变化后必须新建 Generation 和两份完整复审。
