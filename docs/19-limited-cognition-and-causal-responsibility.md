# 19 - 有限认知、因果归属与专业可执行性

## 目标

v3.4 处理一种在表面质感改善后仍然明显的 Agent 写作模式：主角观察到任何细节，
叙述立即给出唯一正确解释，世界随后用对手惊讶、机会出现或谜底暗示证明主角聪明。

本里程碑不要求主角故意犯错。它要求：

1. 事实、人物假设和作者已知彼此分离；
2. 世界与配角拥有独立于主角的目标；
3. 关键条件的责任归属可以追溯；
4. 专业结论具备执行条件和失败方式；
5. 人物的能力通过选择与后果显形，而不是靠术语和旁白背书。

SQLite schema、长篇记忆协议和章节状态链均不改变。新增内容属于 Markdown 规划、
纯函数 narrative gate 与审稿角色职责。

## 四层写前约束

### 故事发动机

正式稿的 `planning/story-engine.md` 从 v3.4 起是 blocking 材料，不能保持空模板，
也不能用“无需”豁免。除欲望、不可逆选择和即时代价外，还必须建立：

- 对手或世界的独立目标；
- 即使主角判断正确，对方仍会采取的行动；
- 主角当前可能错误的模型；
- 推翻该模型所需的证据；
- 一个具体未解承诺。

### 决策摩擦

`1c. 决策问题` 的五项中至少两项必须真实成立：

- 不能同时得到的两样东西；
- 拒绝承认；
- 误读；
- 不能说出口的话；
- 最终接受的具体代价。

章型只校准信息密度、对白配比与语域。立章可以没有终局式人生选择，但不能让
人物在没有拒绝、误读、隐瞒或代价的情况下平滑完成能力展示。

### 认知与可证伪假设

`1d. 认知与可证伪假设` 只登记会推动关键行动的推断：

| 字段 | 责任 |
|---|---|
| 观察事实 | 限知人物真正看见、听见或已被告知的内容 |
| 人物当前假设 | 人物据此形成的解释 |
| 替代解释 | 同一事实还可能意味着什么 |
| 置信度 | 低 / 中 / 高，不使用伪精确百分比 |
| 可推翻证据 | 什么后续现象会证明当前假设不成立 |
| 本章状态 | 未决 / 证实 / 推翻 / 误判 |

没有关键推断的章节可以显式豁免，但必须写具体原因。causal-editor 负责判断豁免
是否诚实；机器门只验证材料存在。

### 因果归属与专业审计

`3c. 因果归属账本` 每个正式章至少一条，记录关键动作或条件的提出/执行者、
对象、当场知情者、来源 beat 与后果承担者。

`5b. 专业判断审计` 用于金融、医疗、法律、刑侦、工程、历史制度和职业手艺：

- 可观察证据；
- 未证前提；
- 本金、权限、工具、时间、对手配合等执行条件；
- 成本与风险；
- 失败或证伪方式。

无专业判断时允许显式豁免。专业术语本身不构成专业正确性。

## Narrative Gate

formal 模式新增 blocking：

- 决策摩擦少于两项；
- 缺少认知账本且没有具体豁免；
- 因果归属账本少于一条；
- 缺少专业审计且没有具体豁免；
- 场景包声明有关键对白，但对白账本不存在或未填写；
- `planning/story-engine.md` 缺失或仍是空模板。

exploration 模式仍跳过这些材料门，保持试声和快速实验能力；探索稿不能进入 ready。

## 审稿责任

- `context-collector`：摘要明确区分 Canon 事实、人物已知、人物猜测和替代解释。
- `causal-editor`：检查全知式推断、责任漂移、世界只证明主角、专业结论不可执行。
- `line-editor`：检查“观察 → 履历/原理 → 正确判断 → 他人惊讶”的能力证明循环。
- `consistency-guard`：防止人物假设被后文静默升级为事实，核对提出者与知情者。
- `chapter-editor`：检查主角是否可能失败、对手是否有独立意志、专业判断是否承担成本。

机器 gate 不判文学优劣，也不能验证某项金融、医疗或法律判断事实上正确。它负责
迫使规划材料暴露前提，语义审稿角色再判断这些前提是否成立。

## 实验证据

本次规则来自三个 Agent demo，而非单篇主观修稿：

- Kimi《拾骨人》：证明物件、规矩和关系能承载专业性，也暴露责任归属漂移；
- Kimi《重回一九九二》：证明强主角可以通过有风险的操作显形，不必依赖解释；
- DeepSeek《大明财阀》：集中暴露全知推断、能力证明循环和专业不可执行问题。

详细证据见 `docs/examples/agent-demo-v34-evidence.md`。这些规则仍应通过后续跨作品、
跨类型和跨模型实验继续验证；语义规则不得仅因单次成功就升级为词句级硬禁令。

## 迁移

存量项目运行：

```cmd
python -m app.novel_forge.skill_adapter --root <ABS_ROOT> sync-tools <slug> --dry-run
python -m app.novel_forge.skill_adapter --root <ABS_ROOT> --confirm sync-tools sync-tools <slug>
```

`sync-tools` 会刷新托管的场景包模板和 Agent 定义，不覆盖已填写的章节场景包。
旧正式章若要重新进入 ready，必须按 v3.4 模板补齐材料并重新审稿。
